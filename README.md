# Data Vault 2.0 Storage Layer: Orders & Customers

Данный репозиторий содержит SQL-скрипты для развертывания слоя хранения данных (**Raw Vault**), спроектированного по методологии Data Vault 2.0. Архитектура ориентирована на масштабируемость, аудит и хранение полной истории изменений.

## 🏗 Структура модели данных

Модель разделена на три типа сущностей для разделения бизнес-ключей, связей и описательных атрибутов.

### 1. Hubs (Хабы)
Хранят уникальные бизнес-ключи сущностей и обеспечивают интеграцию данных из разных источников.
- `Customers_hub`: Реестр клиентов (`customer_id`).
- `Products_hub`: Реестр продуктов (`product_id`).
- `Orders_hub`: Реестр заказов (`order_id`).
- `Order_Statuses_hub`: Справочник уникальных идентификаторов статусов.

### 2. Links (Линки)
Описывают транзакционные связи и отношения между бизнес-сущностями (многие-ко-многим).
- `Orders_Products_link`: Связь заказов с товарами.
- `Orders_Customers_link`: Привязка заказов к конкретным клиентам.
- `Orders_Statuses_link`: Связь заказов с их статусами.

### 3. Satellites (Сателлиты)
Хранят исторические описательные атрибуты (SCD Type 2).
- `Customers_sat`: Персональные данные (имя, email, телефон).
- `Products_sat`: Характеристики товаров (название, категория).
- `Orders_sat`: Детали транзакций (сумма, количество).
- `Order_Statuses_sat`: Текстовое описание статуса.

## 🛠 Технические стандарты

- **Первичные ключи (PK):** Используется тип `UUID` для `hash_key`. Это позволяет генерировать ключи на этапе стейджинга и выполнять параллельную загрузку без ожидания ответа от БД.
- **Бизнес-ключи:** В хабах установлены ограничения `UNIQUE` и `NOT NULL` на исходные идентификаторы.
- **Аудит:** Каждая таблица содержит поля:
    - `load_date`: Время вставки записи.
    - `record_source`: Система-источник данных.
- **Управление изменениями:** В сателлитах используется `hash_diff` (текстовый хэш от набора атрибутов) для эффективного обнаружения изменений в данных (Delta-загрузка).
